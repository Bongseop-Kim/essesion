# Admin UI/UX 감사 후속 구현 (2026-07)

## 범위

`ESSE_SION_admin_UIUX_audit.md`의 우선순위와 완료 기준을 적용했다. 이번 변경은 관리자 프론트의 정보 구조, 편집 안전성, 목록 검색·필터 일관성, 기술 정보의 점진적 공개를 다룬다. 누락된 검색·KST 기간 조회는 admin API 계약과 생성된 `api-client`까지 함께 보강했으며 데이터베이스 스키마는 변경하지 않았다.

## 반영한 항목

### P0 — 조회와 고위험 변경 분리

- 가격 관리는 읽기 전용 가격표가 기본이다. admin만 `가격 수정`으로 편집을 시작하며, 고정 저장 영역에서 변경 건수와 취소/검토 액션을 확인한다. 확정 전 기존값/새 값, 즉시 적용 영향, 변경 사유를 함께 보여준다. operation ID는 요청의 멱등성에는 사용하지만 화면에는 노출하지 않는다.
- 설정은 설정별 읽기 카드가 기본이다. 한 번에 한 설정만 명시적으로 수정하며 적용 범위, 시스템 기본값, 마지막 변경 시각을 표시한다. 초기 토큰 변경은 신규 계정에만 영향을 준다는 경고와 전후 값 검토를 거친다.
- 고객 토큰은 기본 화면에서 입력 폼과 operation ID를 제거했다. `토큰 조정` 다이얼로그 안에서 현재 전체/유료/보너스 잔액, 변경 후 잔액, 유료 우선 회수 규칙, 사유를 검토한 뒤 `토큰 N개 지급/회수`로 확정한다.
- 문의 답변은 읽기 전용이 기본이다. `답변 작성`/`답변 수정`으로 전환하고, 고객에게 보일 답변을 미리 본 뒤 구체적인 등록/수정 액션으로 저장한다.
- 쿠폰 일괄 발급과 회수는 기술 ID 대신 대상 수, 중복 제외 기준, 만료일, 비가역 영향, 사유를 검토 화면에 표시한다. 중복된 쿠폰 수정 액션도 하나로 정리했다.
- 편집 중에는 탭·경로·검색 조건·브라우저 뒤로 가기를 포함한 이탈을 막는다. 선택 시점 revision을 유지하고, 동일 요청 재시도에는 같은 operation ID를 재사용하되 실패 뒤 payload가 바뀌면 새 ID를 발급해 멱등 키 오용을 막는다.

### P1 — 상세 정보 구조와 목록 도구

- 고객 상세는 `개요/주문/쿠폰/토큰`, 주문 상세는 `개요/항목/배송·수선/결제·클레임/활동 이력`, 견적 상세는 `요청 요약/제작 사양/견적 제안/참고 이미지/이력`, 클레임 상세는 업무와 이력 탭으로 분리했다. 탭은 URL로 복원하며 업무 유형에 맞지 않는 배송 탭은 숨긴다.
- 주문, 견적, 클레임의 서버 허용 액션을 상세 정보보다 앞에 배치해 다음 작업을 첫 화면에서 찾을 수 있게 했다.
- 견적 제작 옵션은 알려진 스냅샷 필드를 한국어 설명으로 보여주고 원문만 기술 정보에 둔다. 진행 단계를 텍스트로 표시하고 견적 작성/수정 진입점을 명확히 했다.
- 전체 목록에 공통 적용 필터 바를 추가했다. 활성 필터 수, 개별 제거, `전체 초기화`를 제공하며 기본값인 `전체`는 칩으로 반복하지 않는다.
- 목록 필터는 별도 카드 대신 표 카드 상단의 compact 툴바로 통합했다. 모든 주요 목록은 검색 입력이 가용 폭을 최대한 채우고 오른쪽 고정 폭 `필터 N`과 입력 하단을 맞춘다. 상태·카테고리·유형·세부 분류·기간 같은 선택 조건은 모두 필터 패널로 통일했다. 상세 조건은 데스크톱 SidePanel과 모바일 BottomSheet에서 초안으로 편집하며 `취소`는 폐기하고 `필터 적용`에서 한 번에 반영한다.
- 검색 대상은 업무 식별자에 맞췄다. 주문·클레임·견적은 각 번호, 수기 주문은 고객명·연락처, 상품은 상품명·코드, 쿠폰은 쿠폰명·표시명·쿠폰 ID, 고객은 이름·이메일·연락처, 문의는 제목·내용, 생성 운영은 작업 ID·요청 ID, Motif는 ID·이름·소스, 결제 이상은 operation/request ID와 관련 UUID를 조회한다. 검색어는 서버 조회 조건에 포함해 페이지네이션의 총 건수와 일치한다.
- 모든 주요 목록의 상세 필터에 기간을 제공한다. 기본적으로 KST 생성·등록·발생일을 조회하고, 수기 주문만 업무 의미에 맞게 주문일을 사용한다. 시작일이나 종료일 하나만 입력하면 열린 범위로 조회하며 `전체`, 최근 7일·30일·90일 프리셋과 KST 직접 입력을 함께 제공한다. 상품의 페이지 크기는 필터에서 분리해 페이지네이션 영역에서 20·50·100개를 선택하게 했고, 별도 정렬 선택기는 제거해 표 헤더 정렬만 남겼다.
- 상태/유형/기간/정렬/페이지처럼 민감하지 않은 목록 상태는 URL에 유지한다. 고객 검색어, 요청 ID, source 같은 검색 입력은 메모리에만 두고 전체 초기화 시 입력과 제출 상태를 함께 지운다.
- 페이지네이션은 `현재 범위 / 총 N건`과 페이지 크기를 한 곳에서만 표시한다. 로딩·오류 중에는 이전 총 건수를 `0건`으로 오인하게 표시하지 않고, URL의 페이지가 실제 범위를 넘으면 마지막 유효 페이지로 교정한다.
- 결제 이상은 예상/실제/차이를 먼저 문장으로 요약하고 해결 액션을 기술 정보보다 앞에 둔다. 견적·클레임·생성 상세도 운영자가 읽는 한국어 요약을 먼저 제공하고 raw JSON, 원문 코드와 UUID는 복사 가능한 `기술 정보` 접기 영역으로 옮겼다.

### P2 — 탐색과 운영 피드백

- 데스크톱 사이드바와 모바일 메뉴의 시각적 그룹 제목은 제거하고 1px 약한 Divider로만 구분했다. `운영`, `고객`, `상품·프로모션`, `생성·에셋`, `시스템`은 접근 가능한 section 이름으로 유지하며 상세 경로에서도 현재 메뉴에 `aria-current="page"`를 제공한다.
- 공통 상태 배지는 영문 운영 코드를 한국어 상태명으로 표시하고 텍스트를 함께 제공한다.
- 생성 운영에는 자동 갱신 상태, 마지막 갱신 시각, 일시정지/재개 제어를 제공하면서 비활성 탭·숨김 문서·terminal 상태의 기존 폴링 제한을 유지한다.

## 의도적으로 이연한 항목

다음 항목은 화면만 추가하면 감사 가능성이나 데이터 의미를 보장할 수 없어 이번 프론트 변경에 포함하지 않았다.

- 가격 예약 적용, 가격 버전 비교/롤백: 적용 시점과 버전 읽기 모델, 예약 실행기, 롤백 권한 및 감사 API가 먼저 필요하다.
- 설정 변경 이력과 고객 개인정보 `보기`: actor가 포함된 이력 조회 및 역할별 마스킹/조회 감사 API가 필요하다.
- 저장된 보기, 사용자별 열/밀도: 서버 또는 명시적으로 승인된 사용자 설정 저장 정책과 기본 보기 정의가 필요하다.
- 메뉴 미처리 배지, 문의 SLA 임박, 생성 p50/p95와 지연 저장 보기, 대시보드 예외 큐: 해당 집계와 SLA 기준을 제공하는 읽기 모델이 필요하다.
- 가격 변경 이력, 고객 감사 이력, 긴 타임라인의 서버 페이지네이션: 현재 상세 계약에 별도 조회 경계가 없다.
- 운영자 3~5명 과제 수행 시간과 오류율 비교, WCAG 전문 감사: 배포 가능한 환경과 실제 운영자 세션에서 별도 수행한다.

## 검증

최종 검증 결과는 아래에 기록한다.

- 자동 검증: `pnpm codegen`으로 134-path OpenAPI client를 재생성했다. `pnpm lint`(Biome 462개 파일, harness 포함), 로컬 공개 빌드 환경값을 명시한 `pnpm turbo build typecheck test`(10/10 task), `uv run ruff check .`, `uv run pyright`, `uv run pytest`가 통과했다. 테스트는 admin 185건, shared 64건, store 164건, Python 657건이 모두 통과했다.
- Aside 데스크톱(1440×900): 주문·수기 주문·견적·클레임·결제 이상·문의·고객·상품·쿠폰·생성 작업·Seamless·Motif 12개 보기를 전수 확인했다. 모두 업무별 검색과 필터 버튼, KST 시작일·종료일을 제공했고 검색 폼은 툴바의 약 92%를 사용하면서 84px 필터 버튼과 하단선이 일치했다. 필터는 오른쪽 SidePanel로 열렸고 메뉴는 1px Divider 5개로 구분되며 시각적 그룹 제목 없이 접근 가능한 section 이름을 유지했다.
- Aside 모바일(390×844): Motif 검색 폼이 필터 버튼을 제외한 가용 폭을 채우고 하단 정렬되는 것, 전체 폭 BottomSheet의 범위·최근 기간·KST 날짜 입력, 제목 없는 메뉴 Divider 5개와 의미 그룹 5개를 확인했다. 문서 가로 overflow는 0px였고 콘솔·페이지 오류는 없었다.
- 변경 확정 버튼은 누르지 않아 브라우저 검증 중 서버 데이터 mutation은 발생하지 않았다.
